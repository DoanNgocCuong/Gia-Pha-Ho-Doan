## 2. Luồng xử lý chính (theo code)

1. **Bootstrap**: `bootstrapTree()` → `loadPrintSizeConfig()` → áp config lên CSS (`--node-width`, `--stratum-gap-*`, `@page` in, override đời 1–4 nếu có) → `loadTreeData()` đọc JSON gia phả.
2. **Render**: `renderTreeFromData()` xây nested `<ul>/<li>` và các ô `.node` có `data-node-id`.
3. **Mô hình đồ thị**: `buildStratifiedModel` → `levels[]`, `edges[]`; đời focus = đời đông nhất (`computeBusiestGenerationDepth`).
4. **Layout**: `computeAbsoluteLayout(model, focusDepth)` — Phase 1 (focus đều) → Phase 2 + 2b/2c (tổ tiên) → Phase 3 (hậu duệ cụm quanh cha) → **Phase 3b/3c** (clamp mép phải hậu duệ mirror 2b/2c) → Phase 2d; `applyAbsoluteLayout()` đặt `position: absolute; left/top` và kích thước vùng `.tree`. Chi tiết: [V4_ADR2.md](V4_ADR2.md).
5. **Cạnh**: `drawTreeEdges()` dùng `getBoundingClientRect()` — tương thích với absolute layout; có observer khi resize/zoom.
6. **Chữ trong ô**: `fitNodeText()` / normalize nhãn — font px suy ra từ config typography.

Chi tiết pseudo-code và chỉ số đo: [SDD §3](../SDD.md#31-tổng-quan-thuật-toán).

## Phụ lục A — Logic "Cách B" / bbox-midpoint (✅ ĐÃ TRIỂN KHAI ở Phase 2 v2)

Phần dưới mô tả phương án **tâm cha = midpoint bbox** subtree ở đời focus (`min/max` thay vì `avg` lặp tầng). Đây là hướng giảm bias khi cây lệch; **code hiện tại đã chuyển sang Cách B (Phase 2 v2)** — xem [SDD §10](../SDD.md#10-phase-2-v2--bbox-midpoint-reingoldtilford-tight).

### Một câu

> "Mỗi tổ tiên đứng **chính giữa** dải mà cây con của nó chiếm ở đời focus."

### 3 thành phần

```
┌─────────────────────────────────────────────────────────┐
│  Mỗi node v lưu 2 số:                                  │
│     L(v) = mép TRÁI nhất của cây con v ở đời focus     │
│     R(v) = mép PHẢI nhất của cây con v ở đời focus     │
│                                                         │
│  Tâm của v:                                            │
│     x(v) = ( L(v) + R(v) ) / 2                         │
└─────────────────────────────────────────────────────────┘
```

### Quy trình bottom-up (4 bước)

```
Bước 1.  Đặt đời focus đều nhau (Phase 1, không đổi).
         Mỗi ô v ở đời focus:  L(v) = x(v) - W/2
                                R(v) = x(v) + W/2

Bước 2.  Lên 1 đời (d = focus-1, focus-2, ..., 0).
         Cho mỗi cha p ở đời d:
              L(p) = min(L của tất cả con của p)
              R(p) = max(R của tất cả con của p)
              x(p) = ( L(p) + R(p) ) / 2     ← TÂM = MIDPOINT BBOX

Bước 3.  Sweep phòng đè (giữ nguyên).
         Nếu tâm tính ra "đè" anh trước, đẩy sang phải:
              x(p) = max( x(p), prev_right + G + W/2 )

Bước 4.  Cập nhật L(p), R(p) sau sweep.
         L(p) = min( x(p) - W/2, L(p) cũ )
         R(p) = max( x(p) + W/2, R(p) cũ )
         → để truyền lên đời cao hơn cho đúng.
```

### Ví dụ tay với cây tí hon

Giả sử đời focus có 5 ô đặt tại x = `10, 20, 30, 40, 50`.

```
        d=0:           ROOT?
                       /        \
        d=1:        A              B
                  / | \           / \
        d=2:    a1 a2 a3        b1  b2
                |  |  |          |   |
        focus: 10 20 30         40   50
```

**Đặt focus row** (Bước 1):

```
ô |  x | L  | R
-----------------
10 | 10 | 5  | 15
20 | 20 | 15 | 25
30 | 30 | 25 | 35
40 | 40 | 35 | 45
50 | 50 | 45 | 55
```

**Lên d=1** (Bước 2):

`A` có 3 con `a1, a2, a3` (tức 3 ô focus 10, 20, 30):

```
L(A) = min(5, 15, 25)   = 5
R(A) = max(15, 25, 35)  = 35
x(A) = (5 + 35) / 2     = 20      ← chính giữa cây con của A
```

`B` có 2 con `b1, b2` (40, 50):

```
L(B) = min(35, 45)  = 35
R(B) = max(45, 55)  = 55
x(B) = (35 + 55)/2  = 45          ← chính giữa cây con của B
```

**Lên d=0** (root):

```
L(ROOT) = min(L(A), L(B)) = min(5, 35) = 5
R(ROOT) = max(R(A), R(B)) = max(35, 55) = 55
x(ROOT) = (5 + 55) / 2  = 30      ← chính giữa CẢ CÂY (đúng giữa focus row)
```

**So sánh với cách hiện tại (`avg con trực tiếp`):**

```
Cách hiện tại:
   x(A)    = avg(10, 20, 30) = 20    ✓ trùng nhau
   x(B)    = avg(40, 50)     = 45    ✓ trùng nhau
   x(ROOT) = avg(20, 45)     = 32.5  ✗ lệch phải 2.5

Cách B:
   x(ROOT) = (5 + 55)/2      = 30    ✓ đúng giữa focus row
```

→ Khi cây càng **không đối xứng** (số con trái ≠ phải, độ rộng cây con khác nhau), chênh lệch giữa `avg` và `bbox-mid` càng lớn. Trong cây gia phả thật, sai số này **cộng dồn** lên thành 60–70 cm như đo được ở Cụ Rũng / Cụ Tổ.

### Vì sao công thức `min/max` không bị "cộng dồn"

```
Tính chất TOÁN HỌC quan trọng:

   min( min(a,b), min(c,d) )  =  min(a,b,c,d)
   max( max(a,b), max(c,d) )  =  max(a,b,c,d)
```

Nghĩa là: **min/max là phép toán "phẳng"** — gộp nhiều tầng lại thành 1 tầng cũng cho đúng kết quả. Còn `avg` thì:

```
   avg( avg(a,b), avg(c,d) )  ≠  avg(a,b,c,d)        nếu trọng số khác
```

→ `avg` 2 tầng ≠ `avg` flat của tất cả → bias xuất hiện.

### Kết luận 1 dòng (Cách B)

**Cách B = "dùng `min/max` thay cho `avg`"**, vì `min/max` không bị cộng dồn sai số khi đệ quy lên nhiều tầng. Tâm cha luôn đứng đúng giữa cây con của mình ở đời focus, dù cây có lệch hay không.

*(Triển khai thực tế: [SDD §10](../SDD.md#10-phase-2-v2--bbox-midpoint-reingoldtilford-tight). Đo: root cx 197.25 → 136.81 cm sau khi áp Cách B.)*

## Phụ lục B — Phase 2b: Clamp mép phải per-row theo đời focus

### Bối cảnh

Sau Phase 2 v2, root về đúng tâm nhưng một số đời tổ tiên vẫn trồi sang phải so với mép phải đời focus do bbox của vài nhánh cây con trải rộng:

| Đời (1-based) | depth | rightCm | overflow vs focus |
|---|---|---|---|
| Đời 5 | 4 | 266.18 | +7.44 cm |
| Đời 6 | 5 | 273.62 | +14.88 cm |
| Đời 7 | 6 | 266.62 | +7.88 cm |
| **Đời 8 (focus)** | **7** | **258.74** | **0** |

User: *"đời nào bị trồi ra ngoài bên phải => kéo sang trái toàn bộ đội hình của đời đó"*.

### Công thức

Với mỗi đời `d ∈ [0, focus-1]`:

```
R_focus     = max over focusRow of (x + W/2)
R_d         = max over levels[d] of (x + W/2)
L_d         = min over levels[d] of (x - W/2)
overflow_d  = max(0, R_d - R_focus)
shift_d     = min(overflow_d, max(0, L_d))    // cap: không xuyên x = 0
```

Nếu `shift_d > 0`: với mọi `entry ∈ levels[d]`, `x ← x − shift_d`.

### Tính chất

- **Không sinh chồng lấn**: shift đồng đều toàn bộ node trong cùng đời → khoảng cách trong đời giữ nguyên.
- **Per-row độc lập**: không dây chuyền giữa các đời (mỗi đời shift một lượng khác nhau → mũi tên cha-con có thể bend).
- **Kết quả đo**: tree width 273.63 → 270.13 cm (−3.5 cm); đời 5 còn trồi 11.38 cm vì `leftmost = 0` cap shift.

Chi tiết: [SDD §11](../SDD.md#11-phase-2b--clamp-mép-phải-per-row-theo-đời-focus).

## Phụ lục C — Phase 2c: Cascading left-pack (suffix shift)

### Bối cảnh

Sau Phase 2b, một số đời tổ tiên vẫn trồi phải nhưng `leftmost = 0` đã chặn shift đều — không thể kéo cả đội hình sang trái thêm:

| Đời | depth | overflow sau Phase 2b | leftmost | Có shift đều? |
|---|---|---|---|---|
| Đời 5 | 4 | +3.96 cm | 0 cm | ✗ (cap = 0) |
| Đời 6 | 5 | +11.44 cm | 0 cm | ✗ (cap = 0) |
| Đời 7 | 6 | +7.92 cm | 0 cm | ✗ (cap = 0) |

Tuy nhiên nội bộ các row này có **khe trống lớn** giữa anh em (`maxGap` đời 6 ≈ 957 px ≈ 25 cm) → có thể nén để giảm overflow.

User mô tả: *"check xem ở cuối row có dịch được ko, nếu bị chặn thì check ô đằng trước, ... cho đến 1 ông có thể dịch được đủ số cm cần thì dịch."*

### Logic toán học — vì sao phải SUFFIX shift, không single-node shift

**Quan sát then chốt**: sau Phase 2 v2 + sweep, các khe nhỏ nhất giữa anh em luôn `= G` (ép sát). Khe lớn chỉ xuất hiện tại các vị trí "nhảy" do bbox cây con dài. Vì thế:

- Node cực phải `i = N−1` luôn có gap với `i−2` ≈ `G` (đã sát) ⇒ slack ngay trước nó = 0.
- Nếu chỉ shift node cực phải, `myShift = min(remaining, 0) = 0` ⇒ **overflow không bao giờ giảm**.

⇒ Khi tìm thấy slack tại vị trí `i*`, phải shift **toàn bộ suffix `[i*..N−1]`** sang trái cùng một lượng. Lý do:

1. Các khe trong `[i*+1..N−1]` đều = `G` (sát) → không thể tách rời.
2. Shift suffix giữ tất cả khe phải không đổi, chỉ ép khe `(i*−1, i*)` từ `gap` xuống còn `gap − myShift ≥ G`.
3. Mép phải của row giảm đúng `myShift` cm.

### Pseudo-code

```
remaining ← overflow_d
for i = N−1 down to 0:
    if i = 0:
        slack ← x_0 − W/2                       // tới đụng x = 0
    else:
        gap   ← (x_i − W/2) − (x_{i−1} + W/2)
        slack ← max(0, gap − G)
    myShift ← min(remaining, slack)
    if myShift > 0:
        for j = i to N−1:
            x_j ← x_j − myShift                 // SUFFIX shift
        remaining ← remaining − myShift
    if remaining ≤ 0: break
```

### Kết quả đo

| Đời | depth | Trước Phase 2c | Sau Phase 2c | Δ |
|---|---|---|---|---|
| Đời 5 | 4 | +3.96 cm | **0 cm** | −3.96 |
| Đời 6 | 5 | +11.44 cm | **0 cm** | −11.44 |
| Đời 7 | 6 | +7.92 cm | **0 cm** | −7.92 |
| Tree width | – | 271.59 cm | **263.68 cm** | −7.91 cm |
| Overlap | – | 0 | **0** | OK |

### Trade-off

- **Gap trong row không đều 0.25 cm**: minGap ≈ G, maxGap có thể ~25 cm tại các vị trí "nhảy".
- **Mũi tên cha-con bend mạnh hơn**: vì các đời shift khác nhau và một phần row bị pack.
- **2c chỉ ancestors**; hậu duệ dùng **3b/3c** sau Phase 3 (cùng tinh thần suffix pack). Bảng đo Phụ lục C / SDD §12.5 phản ánh trạng thái trước 3b/3c — cột “Đời 9 +3.52 cm” không còn là hành vi mặc định.

Chi tiết: [SDD §12](../SDD.md#12-phase-2c--cascading-left-pack-suffix-shift-cho-đời-tổ-tiên), [V4_ADR2](V4_ADR2.md).

## Phụ lục D — Edge Routing với Greedy Track Assignment (Channel Routing)

### Bối cảnh

User gửi screenshot cho thấy bus mũi tên xếp chồng thành dải đỏ dày giữa 2 đời. Code cũ đặt `midY = startY + 0.68 * distY` cố định cho mọi cạnh ⇒ bus của các cha có `busInterval` overlap vẽ chồng lên cùng 1 đường ngang.

**Constraint từ user (Mức 1)**: 2 cha cùng đời thì mũi tên không được **chồng đoạn** (cho phép cắt 1 điểm).

### Best practice từ ngành

| Tên | Tác giả/Tool | Áp dụng |
|---|---|---|
| **Channel Routing với Track Assignment** | Soukup, VLSI 1978 | Route dây trong "channel" giữa 2 layer transistor, mỗi dây 1 "track" |
| **Layered Hierarchical Edge Routing** | Graphviz `dot`, ELK Layered | Hierarchical layout với edge merging và track assignment |
| **Edge bundling** | Holten 2006 | Gộp edges cùng hướng |

Phương án Phase B = **Greedy Track Assignment trong Channel Routing** — đúng best practice của Graphviz `dot` và ELK Layered.

### Thuật toán

```
busInterval(p) = [min(p.cx, leftmost child.cx),
                  max(p.cx, rightmost child.cx)]

Với mỗi đời cha d:
  1. Sort các cha theo busInterval.left tăng dần.
  2. lanes = []   (mỗi phần tử = busRight đã đặt cuối cùng)
  3. For each parent p in sorted order:
       for k = 0..lanes.length:
          if lanes[k] + GAP < busInterval(p).left:
             assign p → lane k
             lanes[k] = busInterval(p).right; break
       if no lane found:
          push new lane
          assign p → new lane index
  4. midY(p) = startY + (0.45 + 0.40 * lane(p) / (numLanes-1)) * distY
```

**Tính chất**: 2 cha overlap busInterval → BẮT BUỘC khác lane → khác midY → bus không chồng đoạn.

### Minh hoạ

```
TRƯỚC (mọi cha cùng midY = 0.68 * distY):
  Cha:    [A]    [B]      [C]    [D]   [E]   [F]
           |======|========|======|=====|=====|       ← TẤT CẢ bus cùng độ cao
           |==============|                            ← bus B chèn ngang qua A
                    |======|======|=====|              ← bus C, D, E chồng nhau
          v v v   v v v   v v v  v v v v v v          
  Con:    a1..    b1..    c1..   d1..  e1..  f1..

SAU (greedy track assignment, 2 lane đủ):
  Cha:    [A]    [B]      [C]    [D]   [E]   [F]
           |==|         |===|         |===|             ← lane 0 (A, C, E)
                |===|         |===|         |===|       ← lane 1 (B, D, F)
          v v v   v v v   v v v  v v v v v v          
  Con:    a1..    b1..    c1..   d1..  e1..  f1..
```

### Trade-off

- **Mũi tên bend mạnh hơn**: cha lane cao có bus gần đời con, cha lane thấp có bus gần đời cha. Trunk dài/ngắn khác nhau.
- **Cắt 1 điểm vẫn có thể xảy ra (Mức 2)**: bus cha A có thể CẮT trunk cha B nếu B nằm giữa busInterval của A. Đây là intersection 1 điểm, không chồng đoạn — chấp nhận theo Mức 1.
- **Idempotent**: chạy lại cho cùng kết quả.

### Tính chất bảo toàn

- **Không phá layout node**: chỉ thay `midY` per parent.
- **Không phá hình dạng trunk-bus-drop**: vẫn orthogonal, chỉ stagger Y.
- **Không phá Phase 1, 2 v2, 2b, 2c, 3 của layout**.

## D-13. Stairstep midY stagger per row (2026-05-10)

User cung cấp ảnh tham chiếu (`docs/stagger-dense-area.png`): mũi tên dạng **stairstep/staircase** với mỗi cha có điểm bẻ ở y khác nhau, tạo cảm giác "thang" liên tục thay vì các bus ngang chồng lên nhau.

### Thuật toán đơn giản hoá (`utils/tree-edges-v2.js`)

Thay greedy track assignment bằng deterministic stagger theo `cx`:

```js
// 1) Group parents by y-band (parents at same depth-pair share band)
parentsByBandY = group parents by (cyBot, minChildTop)

// 2) Within each band: sort by parent.cx ASC, assign idx 0..N-1
// 3) midY = cyBot + EPS + (idx+1) / (N+1) * (gap - 2*EPS)
```

**Tính chất**:

- Mỗi cha trong cùng đời ⇒ midY duy nhất, monotone tăng theo `cx`.
- Trunks/buses/drops ở các y khác nhau ⇒ giảm chồng đoạn xuống 0 (segment-overlap = 0).
- Visual: stair pattern khớp ảnh tham chiếu của user (mỗi mũi tên là một bậc thang riêng).
- **Không cần Phase 2d clamp** ⇒ giữ width tree ≤ 270cm canvas (đo: 260.17cm).

### Lý do bỏ Phase 2d cascade

- Phase 2d (clamp `parent.cx ∈ [L, R]` con trực tiếp + cascading bottom-up) yêu cầu sweep theo preorder kết hợp rewind ⇒ tăng width vượt canvas (282cm > 270cm).
- Mặc dù đảm bảo 0 inter-parent crossings về mặt toán học, **visual không đẹp hơn** so với stagger đơn giản (vì các crossings còn lại với stagger là intersection 1 điểm, không gây rối visual).
- Stagger thuận tiến hơn, deterministic, giữ được layout cũ.
